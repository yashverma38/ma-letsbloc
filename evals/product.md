# Product evals

Human-judged. Run with 5 real users before any launch.

## Flow completion

- [ ] P1. User reaches `/cooked` from landing in <2 taps.
- [ ] P2. User understands they need 4 screenshots from the instructions alone (no external help).
- [ ] P3. User finds the Week view on their phone in <30 seconds.
- [ ] P4. User uploads at least 1 screenshot (overview).
- [ ] P5. If user uploads only 1, the product still works end-to-end.
- [ ] P6. If user uploads all 4, the voice note references ≥3 richer data points (pickups, first pickup, notifications, second app).
- [ ] P7. Upload → voice note plays in ≤15 seconds of total wait.
- [ ] P8. User presses share on the result page within 10 seconds of the voice note finishing.

## Drop-off risks

- [ ] D1. 4-screenshot requirement does not kill conversion. Target: ≥60% of uploaders complete ≥1 screenshot. Kill criteria: <40%.
- [ ] D2. `/cooked/manual` fallback has ≥20% of total flow starts (signals real Android/low-literacy segment).
- [ ] D3. If the overview is uploaded but the other three aren't, result still feels complete (no dead placeholders, no "data missing" energy).

## Fallbacks & errors

- [ ] E1. Daily (non-weekly) screenshot still produces a reasonable voice note.
- [ ] E2. Dark-mode screenshot is extracted identically to light-mode.
- [ ] E3. A blurry screenshot produces a friendly error, not a silent fail.
- [ ] E4. Generate failure on slow network returns the user to the upload page with their files still loaded.

## Tribal fit

- [ ] T1. 4 of 5 Gen-Z testers say "damn that's accurate."
- [ ] T2. At least 1 tester forwards the voice note to a friend unprompted during testing.
- [ ] T3. No tester says the tone feels mean. If any do, soften the script bank.
